# Money Tracker — описание проекта

## О чём проект

Money Tracker — простой веб-сайт, где пользователь записывает свои доходы
и расходы, видит баланс за месяц и список последних транзакций.

Это «бумажный блокнот учёта финансов», только в облаке. Открывается
с любого устройства, ничего не теряется, можно поделиться ссылкой
с финансовым консультантом.

## Технический стек

- **Next.js 16** с App Router и TypeScript
- **Tailwind CSS 4** для стилей
- **shadcn/ui** для готовых компонентов (Card, Input, Button, Form)
- **Supabase** — база данных PostgreSQL + Row Level Security
- **`@supabase/ssr`** — современный пакет подключения к Supabase
  (старый `@supabase/auth-helpers-nextjs` — deprecated, не использовать)
- **Server Actions** — основной способ писать бэкенд-логику
  (вместо классических API Routes)
- **Vercel** — хостинг с автодеплоем по `git push`

## Главные функции

### 1. Просмотр транзакций (READ)
- Главная страница `/` показывает таблицу со всеми транзакциями
- Колонки: дата, тип (доход/расход), категория, описание, сумма
- Цветовая индикация: доходы зелёным, расходы красным
- Если транзакций нет — пустое состояние с CTA «Добавить первую»

### 2. Баланс и сводка
- Над таблицей — три карточки с цифрами:
  - **Доходы за месяц** (сумма всех `income`)
  - **Расходы за месяц** (сумма всех `expense`)
  - **Баланс** = доходы − расходы
- Цифры с разделителями тысяч: «65 000 ₽»

### 3. Добавление транзакции (CREATE)
- Кнопка «+ Добавить» над таблицей открывает форму (можно `<dialog>` или отдельная страница)
- Поля формы:
  - Тип — radio: «Доход» / «Расход» (обязательно)
  - Сумма — number, минимум 1 (обязательно)
  - Категория — select из списка: «Зарплата», «Фриланс», «Еда»,
    «Транспорт», «Развлечения», «Прочее» (обязательно)
  - Описание — text, до 280 символов (необязательно)
  - Дата — date, по умолчанию «сегодня» (обязательно)
- Кнопка «Сохранить» вызывает Server Action `addTransaction`
- Валидация через `zod` (на сервере и на клиенте)
- После успеха — страница обновляется через `revalidatePath('/')`

### 4. Редактирование транзакции (UPDATE)
- Клик по строке таблицы открывает ту же форму, но с заполненными данными
- Server Action `updateTransaction(id, data)`
- После успеха — `revalidatePath('/')`

### 5. Удаление транзакции (DELETE)
- В каждой строке — кнопка-иконка «корзина»
- При клике — confirm-диалог «Точно удалить?»
- Server Action `deleteTransaction(id)`
- После успеха — `revalidatePath('/')`

### 6. Фильтрация (опционально, бонус)
- Кнопки-фильтры над таблицей: «Все», «Только доходы», «Только расходы»
- Реализуем через URL search params (`?type=income`)
- Server Component читает параметр и фильтрует запрос к Supabase

## Структура базы данных

### Таблица `transactions`

| Поле          | Тип               | Обязательное | Описание                            |
| ------------- | ----------------- | ------------ | ----------------------------------- |
| `id`          | `int8`, PK, auto  | Да           | Уникальный номер записи             |
| `amount`      | `numeric(10,2)`   | Да           | Сумма (с копейками)                 |
| `type`        | `text`            | Да           | `"income"` или `"expense"`          |
| `category`    | `text`            | Да           | Категория (Еда, Транспорт и т.д.)   |
| `description` | `text`            | Нет          | Свободный комментарий, до 280 симв. |
| `date`        | `date`            | Да           | Дата транзакции                     |
| `created_at`  | `timestamptz`     | Да, auto     | Когда запись создана в БД           |

### Row Level Security (RLS)

Для учебного проекта (без авторизации) включаем RLS и создаём политику:
- `SELECT`: разрешено всем (учебный проект)
- `INSERT`/`UPDATE`/`DELETE`: разрешено всем (учебный проект)

⚠️ В реальном проекте политики должны проверять `auth.uid() = user_id` —
но авторизацию мы делаем в Модуле 7. Пока — открытый доступ.

## Этапы разработки

1. **Настройка Supabase** — создать проект, забрать ключи, поставить
   `@supabase/ssr`, создать клиентов (browser + server) и middleware
2. **Таблица БД** — создать `transactions` в Supabase через визуальный
   редактор, включить RLS, добавить 5 тестовых записей
3. **Чтение данных** — Server Component на `/` читает транзакции прямо
   из Supabase, выводит в таблицу
4. **Баланс** — компонент `BalanceSummary` считает доходы/расходы/баланс
   и показывает три карточки
5. **Server Action для добавления** — функция `addTransaction` в
   `app/actions.ts`, форма с `<form action={addTransaction}>`,
   валидация через zod, `revalidatePath`
6. **Server Action для удаления** — функция `deleteTransaction(id)`,
   кнопка «корзина» в строке таблицы, confirm-диалог
7. **Server Action для редактирования** — функция `updateTransaction`,
   та же форма, но с предзаполненными значениями
8. **Альтернативный путь — API Route** — параллельно делаем
   `GET /api/transactions` (классическое REST API), чтобы понять
   разницу между двумя подходами
9. **Тестирование** — Thunder Client прогоняем все CRUD через API Route,
   проверяем коды ответов
10. **Деплой на Vercel** — env-переменные Supabase в Vercel, push на main,
    сайт оживает на `*.vercel.app`

## Чего НЕ делаем в этом модуле

- **Авторизация и регистрация** — это Модуль 7. Сейчас приложение
  открыто всем по ссылке.
- **Графики и аналитика** — будут позже, на старте достаточно цифр.
- **Категории как отдельная таблица** — сейчас храним как строку.
- **Загрузка чеков (фото)** — пока без этого.
- **Мобильное приложение** — только веб.

## Дополнительные правила

- Все формы валидируем через `zod` — типы общие для клиента и сервера
- Server Actions используем по умолчанию; API Routes — только если
  нужен внешний клиент (например, мобильное приложение в будущем)
- Никаких `any` в TypeScript
- Никогда не вызывать Supabase Service Role Key из браузера
- Все суммы храним как `numeric`, в коде — `number`. Никаких строк
  для денежных значений
