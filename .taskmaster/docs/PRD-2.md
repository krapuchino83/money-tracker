# Аутентификация для Money Tracker — описание фичи

## О чём фича

Сейчас Money Tracker открыт всем: кто угодно по ссылке видит все
транзакции. Это нормально для учебного прототипа, но не для реального
приложения. Добавляем **аутентификацию** — вход и регистрацию — чтобы:

- у каждого пользователя был свой аккаунт;
- каждый видел **только свои** транзакции, а не чужие;
- незалогиненный посетитель не мог попасть на страницы с данными.

## Технический стек (май 2026)

- **Next.js 16** (App Router) + TypeScript — уже есть из модуля 6
- **Supabase Auth** — встроенная в Supabase система входа/регистрации
- **`@supabase/ssr`** — пакет подключения (старый `@supabase/auth-helpers-nextjs`
  deprecated, не использовать)
- `createBrowserClient` — клиент Supabase для браузера
- `createServerClient` — клиент Supabase для сервера
- **middleware** (`src/middleware.ts`) — обновляет сессию на каждом запросе
- **Row Level Security (RLS)** — политики базы данных, чтобы пользователь
  физически не мог прочитать чужие строки

## Главные функции

### 1. Регистрация по email
- Страница `/register` с формой: email + пароль + подтверждение пароля
- Валидация через zod (пароль ≥ 8 символов, пароли совпадают)
- Вызов `supabase.auth.signUp({ email, password })`
- После регистрации — письмо с подтверждением на email
- Показ сообщения «Проверьте почту»

### 2. Вход в систему
- Страница `/login` с формой: email + пароль
- Вызов `supabase.auth.signInWithPassword({ email, password })`
- При успехе — редирект на `/` (дашборд транзакций)
- При ошибке — понятное сообщение под формой
- Кнопка «Выйти» в шапке — `supabase.auth.signOut()`

### 3. Social Login (Google и GitHub)
- На страницах `/login` и `/register` — кнопки «Войти через Google»
  и «Войти через GitHub»
- Вызов `supabase.auth.signInWithOAuth({ provider })`
- Обработка callback в `src/app/auth/callback/route.ts`
- Провайдеры настраиваются в дашборде Supabase (Authentication → Providers)

### 4. Защита страниц
- `src/middleware.ts` обновляет сессию пользователя на каждом запросе
  (паттерн `updateSession` через `createServerClient`)
- Страницы с транзакциями (`/`, `/dashboard`) доступны только
  залогиненным — иначе редирект на `/login`
- Страницы `/login` и `/register` для уже залогиненного — редирект на `/`

### 5. Связь пользователя с данными
- В таблицу `transactions` добавляется колонка `user_id` (uuid,
  ссылка на `auth.users`)
- При создании транзакции `user_id` берётся из текущей сессии
  (`supabase.auth.getUser()`), не из формы
- На дашборде показываются только транзакции текущего пользователя

### 6. RLS-политики на таблице `transactions`
Заменяем «открытые всем» политики из модуля 6 на настоящие:
- `SELECT`: `auth.uid() = user_id` — вижу только свои
- `INSERT`: `auth.uid() = user_id` — могу создать только свою
- `UPDATE`: `auth.uid() = user_id` — могу менять только свои
- `DELETE`: `auth.uid() = user_id` — могу удалять только свои

## Этапы разработки (станут задачами Task Master)

1. Включить Email-провайдер в дашборде Supabase, проверить настройки
2. Убедиться, что `@supabase/ssr` установлен, клиенты client/server на месте
3. Создать `src/middleware.ts` — обновление сессии (`updateSession`)
4. Сделать страницу регистрации `/register` + Server Action `signUp`
5. Сделать страницу входа `/login` + Server Action `signIn` + кнопку «Выйти»
6. Подключить Social Login (Google, GitHub) + callback-роут
7. Защитить страницы с транзакциями через middleware (редирект на `/login`)
8. Добавить `user_id` в таблицу `transactions`, обновить Server Actions,
   переписать RLS-политики на `auth.uid() = user_id`

## Чего НЕ делаем в этой версии

- **Сброс пароля по SMS** — только email-восстановление, и то позже
- **Двухфакторная аутентификация (2FA)** — продвинутая тема
- **Кастомные email-шаблоны** — используем дефолтные письма Supabase
- **Роли и права (admin/user)** — все пользователи равны
- **Профиль пользователя** (аватар, имя) — отдельная фича на потом

## Дополнительные правила

- Пароли и сессии **никогда** не логируем и не храним вручную — этим
  занимается Supabase
- `user_id` всегда берём из серверной сессии (`getUser()`), **никогда**
  из данных формы — иначе пользователь сможет подделать чужой id
- Все формы аутентификации валидируем через zod
- Service Role Key Supabase **никогда** не вызываем из браузера
- После любой операции, меняющей данные — `revalidatePath('/')`
