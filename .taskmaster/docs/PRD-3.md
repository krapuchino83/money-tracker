# Роли и администратор — описание фичи

## О чём фича

После внедрения аутентификации (модуль 7) все пользователи **равны**: каждый видит только свои транзакции. Нужен **минимальный раздельный доступ**:

- обычный **пользователь** — как сейчас: только свои данные;
- **администратор** — один или несколько доверенных аккаунтов, которым нужен **обзор всех зарегистрированных пользователей** и **действия модерации**: блокировка и удаление учётных записей.

Идентификация администратора — по **email**, заданному в конфигурации приложения. Вы указываете в env **тот же email**, с которым регистрируетесь в приложении (например `you@example.com`); после входа этот пользователь считается админом. Это явное правило конфигурации, **не** приглашение по отдельной «ссылке» — вы просто задаёте env и заходите под этим аккаунтом.

## Цели

1. Видеть список всех пользователей, которые регистрируются в приложении (как минимум: email, дата регистрации, статус блокировки).
2. **Блокировать** пользователя (запретить вход; через Supabase Auth Admin API).
3. **Удалять** пользователя (удаление учётной записи Auth и согласованная очистка связанных данных — см. ниже).

## Не входит в эту версию (v1)

- Роли кроме `user` и `admin` (модератор и т.д.).
- Назначение админа произвольному пользователю через UI без изменения конфигурации/деплоя.
- Аудит-лог действий админа.
- Рассылки писем пользователям из админки.

## Технический стек (май 2026)

- **Next.js 16** (App Router), TypeScript — как в проекте.
- **Supabase Auth** — источник правды по пользователям.
- **Service Role Key** — только на **сервере** (Server Actions, Route Handlers); **никогда** в `NEXT_PUBLIC_*` и не в браузерный бандл.
- **`@supabase/ssr`** — anon-клиент для обычных сценариев; для Admin API — **отдельный** серверный клиент с `SUPABASE_SERVICE_ROLE_KEY` (задокументировать в `.env.example`).
- **RLS** у `transactions` не ослаблять для «чтения всех строк»; список пользователей идёт **только** через сервер + Admin API, не через открытые политики на `auth.users`.

## Модель ролей

- Таблица **`profiles`**: `user_id uuid PK references auth.users`, `role text` с ограничением `('user','admin')`, по умолчанию `'user'`; опционально триггер при создании пользователя в Auth.
- **Админ по email из env:** переменная **`ADMIN_EMAIL`** (один адрес; сравнение после `trim` и приведения к lower case). При входе, если email сессии совпадает, в `profiles` выставляется `role = 'admin'` (или проверка только по env без таблицы на первом этапе — в реализации предпочесть **profiles + env**).
- **Несколько админов (опция):** `ADMIN_EMAILS` через запятую — описать в реализации при необходимости.

Каждое действие админки на сервере: проверить сессию и то, что пользователь — админ (через `profiles` и/или `ADMIN_EMAIL`).

## Функции администратора

### 1. Раздел «Пользователи»

- Маршрут только для админа, например **`/admin/users`**.
- Обычный пользователь и гость **не** получают доступ (редирект на `/` или `404` — выбрать в реализации; не светить существование админки).
- Таблица: **email**, **дата регистрации**, **заблокирован** (да/нет).
- Данные — только серверно (Server Component + Admin API с service role).

### 2. Блокировка

- Кнопки «Заблокировать» / «Разблокировать» по `user_id`.
- Нельзя заблокировать **себя** (админа), чтобы не потерять доступ случайно.
- Реализация: методы бана/разбана в Supabase Auth Admin API (актуальные имена методов см. в документации проекта).

### 3. Удаление

- Кнопка «Удалить» + обязательное подтверждение в UI.
- Удаление пользователя в Auth; транзакции с `user_id` должны каскадно удаляться, если в миграции задан `ON DELETE CASCADE` (согласовать с модулем 7).
- Нельзя удалить **свой** аккаунт админа из UI (или только с дополнительным подтверждением — зафиксировать одним способом).

## UI / UX

- Confirm на удаление; понятные ошибки без утечки секретов.

## Middleware

- Пути `/admin/*`: минимум требование авторизации; проверка роли админа — в **server layout** или **page** (надёжнее, чем только middleware).

## Безопасность

1. Service role — только сервер.
2. Мутации с **zod** и проверкой admin.
3. `ADMIN_EMAIL` и service role не в клиентских env.

## Этапы разработки (для Task Master)

1. Миграция `profiles` + роль по умолчанию `user`.
2. Env: `ADMIN_EMAIL`, `SUPABASE_SERVICE_ROLE_KEY` + `.env.example`.
3. `lib/supabase/admin.ts` (только сервер).
4. Синхронизация роли `admin` для email из `ADMIN_EMAIL`.
5. `/admin/users` — список.
6. Server Actions: бан, разбан, удаление.
7. Проверки: не-админ не видит `/admin/users`; админ видит список и выполняет действия на тестовом пользователе.

## Критерии приёмки

- [ ] Доступ к `/admin/users` только у учётной записи с email = `ADMIN_EMAIL` (и подтверждённой сессией Auth).
- [ ] Список соответствует пользователям проекта в Supabase.
- [ ] Блокировка/разблокировка и удаление работают; каскад по данным согласован с БД.
- [ ] Сборка проходит; service role не попадает в клиент.
