# Редизайн и четыре варианта оформления — описание фичи

## О чём фича

Пользователь может выбрать **одну из четырёх визуальных тем** оформления приложения. Переключение делается через **иконку солнышка** в интерфейсе (единая точка входа): каждое нажатие **циклически** меняет тему в заданном порядке. Темы должны быть узнаваемыми и **не ломать** читаемость данных (таблицы транзакций, формы, балансы).

## Цели

1. Сохранить **текущий светлый вид** как первый режим (база и референс UX).
2. Добавить полноценную **тёмную тему** с корректными контрастами и фокусом доступности.
3. Добавить **тему в духе Fallout** (постапокалиптический терминал / Pip-Boy: зелёный на тёмном, моноширинный акцент, «сканирующие» линии или лёгкий CRT-шум — умеренно, без усталости глаз).
4. Добавить **мультяшную тему в духе Mario** с заметными, но не мешающими **анимациями** и «приколами» (микровзаимодействия, праздничные состояния, иконки/декор — без ущерба производительности и без спама движением).

5. Запоминать выбор пользователя (**localStorage** и/или `prefers-color-scheme` только как **начальное** значение для светлой/тёмной, если нет сохранённого выбора — уточнить в реализации).

## Не входит в v1

- Отдельный **кастомайзер** со слайдерами цветов.
- Разные темы для **разных страниц** в одной сессии.
- Полная смена **верстки** (другая сетка, другой IA) — только визуальный слой и допустимые локальные украшения (бордеры, фоны блоков).
- Локализация текстов тем (тексты UI остаются как в приложении).

## Порядок тем по иконке солнышка

Цикл **по клику** (или по Enter/Space если кнопка в фокусе):

1. **Light** — как сейчас (белая/светлая тема, текущие токены shadcn/Tailwind).
2. **Dark** — тёмная нейтральная тема.
3. **Fallout** — стилизованный «терминал»/Pip-Boy.
4. **Mario** — яркая мультяшная тема + анимации.

После четвёртой — снова **Light**.

**Иконка:** базово солнышко; при необходимости в реализации допускается смена иконки по активной теме (космос/луна/звезда) — но **primary control** остаётся одной кнопкой в одном месте, описанной в этом PRD как «иконка солнышка».

## Технический стек (май 2026)

- **Next.js 16** (App Router), **TypeScript strict**, **Tailwind 4**, **shadcn** (base-nova) — как в проекте.
- **CSS variables** (`:root` / `[data-theme="…"]`) для цветов, радиусов, шрифтов — **один источник правды** для всех тем.
- Клиентский **ThemeProvider** (React context) или эквивалент: хранение `themeId`, применение атрибута к `<html>` или `<body>`, синхронизация с `localStorage`.
- **Без миграций БД** для этой фичи.

### Рекомендуемая модель токенов

- Общие семантические токены: `--background`, `--foreground`, `--card`, `--primary`, `--border`, `--muted`, `--destructive` и т.д. (согласовать с существующими переменными shadcn).
- Для **Fallout** и **Mario** — переопределение тех же токенов + опциональные `--theme-accent`, `--theme-glow`, `--font-display`.
- Шрифты: для Fallout — **один** моноширинный акцент (заголовки или цифры); для Mario — **один** round/playful display при сохранении читаемости основного UI-шрифта для таблиц.

### Производительность и доступность

- **Mario:** анимации через `prefers-reduced-motion: reduce` — отключать или сильно упрощать.
- Контраст текста и интерактивных элементов — ориентир **WCAG AA** для Light/Dark; для стилизованных тем — максимально близко, критичные числа (суммы, баланс) **всегда** читаемы.
- Не блокировать гидратацию из-за «мигания» темы: дефолт в HTML до первого клиентского чтения `localStorage` — паттерн с suppress flash (inline script или известный подход проекта).

## UI / UX по темам

### 1. Light (как сейчас)

- Визуально совпадает с текущим продом после внедрения токенизации.
- Никаких регрессий по отступам и компонентам.

### 2. Dark

- Нейтральный тёмный фон, приглушённые карточки, чёткие границы.
- Состояния hover/focus видимы; скроллбары и диалоги согласованы с темой.

### 3. Fallout

- Тёмный фон, **фирменный зелёный** акцент (не кислотный во весь экран).
- Тонкие декоративные элементы: линии сетки, лёгкий vignette, опциональный шум (CSS/SVG) с низкой непрозрачностью.
- Кнопки и инпуты — «технический» вид (рамки, псевдо-LED для важных статусов при необходимости).
- Звук и видео — **не требуются**.

### 4. Mario (мультяшная)

- Яркая палитра (красный/жёлтый/голубой акценты), округлости, тени-карикатура **умеренно**.
- **Микроанимации:** hover на кнопках, «пружина» для FAB/основного CTA (если есть), confetti или звёздочка при **успешном сохранении транзакции** (опционально v1).
- **Пасхалки:** смена курсора на тематический только в безопасных зонах **или** лёгкий параллакс декора в шапке — не мешать кликам по таблице.
- Анимации — CSS-first; тяжёлые canvas/WebGL **не** входят в v1.

## Критерии приёмки

- [ ] Одна кнопка с **иконкой солнышка** переключает темы строго в порядке Light → Dark → Fallout → Mario → Light.
- [ ] Выбор темы **сохраняется** между визитами (минимум `localStorage`).
- [ ] Все основные экраны (главная с таблицей, диалоги формы, auth-страницы если есть в лейауте) **корректно отображаются** во всех четырёх темах.
- [ ] Нет FOUC/мигания **заметного** для пользователя при загрузке (в пределах разумного для SSR+клиентского хранилища).
- [ ] С **`prefers-reduced-motion`** анимации Mario-слоя отключены или минимальны.
- [ ] Линтер и типы без регрессий; нет утечки секретов (фича только UI).

## Риски и смягчение

| Риск | Митигация |
|------|-----------|
| Перегруз визуалом Fallout/Mario | Строго семантические токены; декор отдельным слоём |
| Плохая читаемость чисел | Отдельный класс/токен для «money» всегда высокого контраста |
| CLS из-за шрифтов | `font-display: swap` + ограничить смену семейств |

## Метрики успеха (качественно)

- Пользователи замечают темы и переключают их без обращения к документации.
- Нет жалоб на «невозможно прочитать баланс» в любой теме.

## Краткий чеклист для разработки

1. Ввести `themeId`: `light` | `dark` | `fallout` | `mario`.
2. Подключить провайдер и кнопку-солнышко в **общий layout** (header/toolbar).
3. Описать токены в CSS и маппинг в Tailwind (как принято в проекте).
4. Пройтись по ключевым компонентам (`transactions-board`, диалоги, таблица) на предмет хардкода цветов.
5. Добавить тестирование вручную + e2e опционально (снапшот наличия `[data-theme]`).
